# Research: Auto-Merge Agent (Issue #97)

## Overview
The Auto-Merge Agent is designed to streamline the development cycle by automatically merging "High-Confidence" Pull Requests. This reduces the latency between code completion and integration, allowing the Society to grow faster while maintaining strict quality standards.

## 1. Safety Gates
Safety gates are non-negotiable conditions that must be met before any merge operation is attempted.

- **CI/CD Pass**: All GitHub Action status checks must be in a `success` state. This includes linting (Ruff, MyPy), unit tests, and integration tests.
- **No Merge Conflicts**: The PR must be in a clean state, mergeable with the target branch (`main`).
- **Reviewer Approval**:
    - For agent-generated PRs: Requires at least one approval from the `critic` role (e.g., Galadriel).
    - For critical system changes: May require human (Jord) approval regardless of confidence score.
- **No Unresolved Conversations**: All comments in the PR must be resolved or acknowledged.
- **Linear History**: The agent should prefer squash-merges to keep a clean, linear history.

## 2. High-Confidence Merge Criteria
High-confidence PRs are those that present minimal risk to system stability.

- **Documentation Changes**: PRs that only modify `.md` files or docstrings.
- **Automated Refactors**: PRs generated by other agents (e.g., linting fixes) that do not change logic and pass all tests.
- **Small Scope**: PRs with a low number of lines changed (e.g., < 20 lines) and high test coverage.
- **Dependency Updates**: Routine updates to non-core dependencies that pass all existing test suites.
- **Self-Healing Patches**: PRs created to fix a known, failing test where the fix is verified by the CI.

## 3. CI Integration Requirements
The agent must interact deeply with GitHub's CI/CD ecosystem.

- **Status Check Polling**: The agent needs to use the GitHub API to check the `status` or `check_runs` of the latest commit in a PR.
- **Event-Driven vs. Polling**: While polling is easier to implement, an event-driven approach (using GitHub Webhooks) is preferred for efficiency.
- **Wait for Completion**: The agent must not attempt a merge while CI is `pending`.
- **Handling Flaky Tests**: If a CI run fails, the agent should not merge and instead post a comment or flag the failure for investigation.

## 4. Implementation Strategy
- **Agent Identity**: A new agent (e.g., Boromir or Samwise) could be tasked with this, or it could be a sub-module of the `Critic`.
- **Tools Used**:
    - `list_prs`: To find candidates.
    - `read_pr`: To inspect files, comments, and status checks.
    - `approve_pr`: To provide the final sign-off if criteria are met.
    - `merge_pr`: To execute the squash-merge.
- **Audit Log**: Every auto-merge must be logged in `memory/shared/journal/` with a clear rationale and a link to the merged PR.

## 5. Risk Mitigation
- **Kill Switch**: A mechanism for Jord or the Critic to disable auto-merging globally if an issue is detected.
- **Merge Window**: Restricting auto-merges to certain times or under human supervision during the initial rollout.
- **Dry-Run Mode**: Initial implementation should log what it *would* have merged without actually performing the operation.
